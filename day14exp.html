Servos, Controller Port & the Benchtop Closed Loop

Actuation, then the moment of truth: tilt the vehicle and watch the gimbal push back.

DAY14

Objectives

  • Drive hobby servos with the Servo library / PWM; map gimbal angle δ ↔ servo microseconds, including linkage geometry, direction, and software travel limits that protect the mount.
  • Port the Day 7 discrete PID — identical gains, sample rate, saturation, and anti-windup — from Simulink into C.
  • Close the loop on the bench: mount the flight computer on (or with) the TVC assembly, command "vertical," and demonstrate that tilting the assembly drives a correcting gimbal deflection in the right direction, promptly, without oscillation.
  • Compare against sim: apply an attitude step on the bench, log the response, and overlay it on the equivalent Simulink run. Explain the differences (servo lag? linkage slop? estimator lag? loop jitter?).
Primer Every piece is now in place — a scheduled 100 Hz loop (Day 11), clean characterized sensors (Day 12), and a validated real-time estimator (Day 13) — except the one that turns knowledge into action. Today adds actuation and closes the loop for real. Driving the servo is mostly geometry: mapping the gimbal angle δ your controller commands to a PWM pulse width in microseconds, respecting the actual linkage direction and travel limits so software can't command the mount into itself. The control law itself should not be reinvented — port the exact discrete PID from Day 7, same gains, same sample rate, same saturation and anti-windup logic, because the entire value of this exercise is a fair, like-for-like comparison between simulated and physical behavior. When you finally mount the flight computer with the TVC assembly and tip it by hand, the gimbal correcting promptly, in the right direction, without oscillating, is the payoff of every day since Day 1. It almost never matches simulation exactly on the first try, and that's expected, not a failure: servo lag, linkage slop, estimator lag, and loop jitter each leave a distinct fingerprint on a step-response overlay, and diagnosing which one you're looking at — using the log data, like an engineer, not a guess — is exactly the skill MR-3 is designed to test.

Concept 1 — Driving a servo: geometry, not control

A hobby servo is a closed-loop positioner in its own right — you command an angle and its internal electronics drive there. You command that angle with a PWM signal whose pulse width (conventionally ~1000–2000 µs, ~1500 µs at center) maps to servo position. The Arduino Servo library hides the raw PWM and lets you write microseconds or degrees. So driving the gimbal is a mapping problem, not a control problem — but the mapping has several real subtleties:

  • Angle-to-microseconds calibration. The controller thinks in gimbal angle δ (radians or degrees); the servo thinks in microseconds. You need the actual mapping for your servo and linkage — it's roughly linear but the slope and center offset are specific to the build. Measure it, don't assume the nominal.
  • Linkage geometry and direction. The servo horn connects to the gimbal through a mechanical linkage that has its own ratio and, critically, a direction. A positive servo command might produce a negative gimbal deflection depending on how the linkage is arranged — the actuator twin of Day 12's sensor sign check. Get this backwards and the loop drives the wrong way just as surely as a flipped gyro.
  • Software travel limits. The physical mount has a hard range (±5–10°, Day 4). Clamp the commanded microseconds in software below the mechanical limit so the controller can never drive the servo into the stop — that protects the hardware and matches the saturation you modeled in sim.
  • Center and trim. "Zero gimbal" is a specific microsecond value you establish by calibration, not necessarily 1500 µs. Find it so that neutral command means thrust-along-axis.

Concept 2 — Port the controller exactly; don't redesign it

The temptation on hardware is to "tweak" the controller. Resist it. The entire scientific value of today is a like-for-like comparison: the same discrete PID you validated in sim, running on hardware, so that any difference in behavior is attributable to the physical effects (servo, linkage, real sensor, real timing) rather than a different controller. Port from Day 7 verbatim:

  • Identical gains — the frozen vehicle_params.m values from Day 10.
  • Identical sample rate — 100 Hz, the loop period you're already holding (Day 11).
  • Identical saturation — the same ±limit, now also enforced by the servo travel clamp.
  • Identical anti-windup — the same clamping/back-calculation logic from Day 7, which matters more on hardware because the servo really does saturate.
  • Derivative on measurement, not error — the Day 6/7 structural choice that avoids derivative kick and tames noise; carry it over exactly.

If the ported controller misbehaves, that's information about the sim-hardware gap, which is the deliverable — not a cue to start hand-tuning. Change one thing only if forced, and log why. A faithful port that reveals a discrepancy is worth far more than a re-tuned one that hides it.

Concept 3 — Bring-up order: open loop before closed loop

Closing a feedback loop on an unstable-by-design system for the first time is exactly when a wiring or sign error becomes violent, so sequence the bring-up to fail safe:

  • Servo checkout, sensors out of the loop. First, with no feedback, command the servo directly: sweep it across its range, confirm the travel limits, find center, and verify the linkage direction (positive command → which way does the gimbal actually point?). This isolates the actuator mapping completely before any feedback can amplify a mistake.
  • Then close the loop. Only once the servo mapping is proven do you connect estimator → controller → servo. Do it ready to cut power, because a sign error now produces immediate divergence.
  • Tilt test. Command "hold vertical," then tip the assembly by hand. The gimbal should deflect to push back toward vertical — correct direction, sensible magnitude, and settling without oscillation. This is the payoff moment, and it's also a complete end-to-end sign check: sensor → estimator → controller → linkage → gimbal, all correct, or it visibly isn't.

If it oscillates or diverges, reach for the Day 10 suspect list rather than random tweaks — and you have logs (Day 12) to diagnose from. A 2 Hz oscillation, a wrong-direction correction, and a slow divergence each point at different causes.

Concept 4 — The sim-vs-hardware overlay: reading the fingerprints

The intellectual core of MR-3 is the step-response overlay: apply the same attitude step on the bench that you applied in sim, log the response, and plot the two on the same axes. They won't match perfectly, and the way they differ is diagnostic — each physical effect leaves a recognizable fingerprint:

  • Servo lag / rate limit → the bench response rises more slowly than sim, or rounds off sharp commands. The actuator can't keep up with what the controller asks.
  • Linkage slop (backlash) → a small dead zone around direction changes; the gimbal doesn't respond until the mechanical slack is taken up. Shows as a flat spot or hysteresis near zero.
  • Estimator lag → the whole response is delayed by roughly the estimator's effective latency; the controller is acting on slightly old attitude.
  • Loop jitter → irregular, noisy timing in the response that a clean-clock sim never shows; if present, revisit Day 11's timing telemetry.
  • Un-modeled friction / mass → the bench assembly's pivot friction and real inertia differ from the model, shifting damping and settling.

The skill being graded isn't "make them match" — it's explaining the difference from evidence in the log. "The bench lags sim by ~40 ms, consistent with our measured servo response plus one cycle of estimator latency" is an engineering answer; "hardware is just different" is not. This is also where you identify the single model improvement that would most reduce the gap — the MR-3 question.

Rehearse the demo today, not tomorrow MR-3 is a live demo, and live demos fail in ways bench testing doesn't: a loose connector, a flat battery, a laptop that won't enumerate the serial port, an SD card that fills up. Run the entire MR-3 sequence end-to-end at least once today — power-on, calibrate, close the loop, do the tilt demo, capture the step response, save the logs — exactly as you'll run it tomorrow. Anything that's going to go wrong is far cheaper to discover now, with a day of margin, than in front of reviewers. A demo that's been rehearsed once is a categorically different thing from one assembled the morning of.

  • Close the loop. Do the tilt test: command vertical, tip by hand, confirm prompt correct-direction correction without oscillation. Debug with the Day 10 suspect list, in the log data.
  • Capture the sim-vs-hardware step-response overlay and write the evidence-based explanation of every visible difference.
  • Rehearse the MR-3 demo end-to-end once today.
Common pitfalls
  • Closing the loop before servo checkout. A sign or limit error is violent under feedback on an unstable plant. Prove the servo mapping open-loop first.
  • Linkage direction backwards. The actuator version of the Day 12 sign bug — the gimbal corrects the wrong way and the vehicle fights itself. Verify direction during open-loop checkout.
  • Re-tuning instead of comparing. Hand-tuning on the bench destroys the like-for-like comparison that is the whole point. Port exactly; treat discrepancies as data.
  • No software travel limit. Without a clamp below the mechanical stop, the controller can drive the servo into the mount and strip a gear or bind the gimbal.
  • Debugging without logs. "It oscillated" is not diagnosable; the logged sensor/estimate/command/servo traces are. Log every run (Day 12).
  • Assembling the demo the morning of MR-3. Rehearse today. Live demos punish the un-rehearsed.
