Turn yesterday's equations into a running simulation, and prove to yourself it's right before anyone asks you to control it.

DAY05

Objectives

  • Implement the 3DOF EOM in Simulink — either from integrator blocks/MATLAB Function blocks, or using the Aerospace Blockset 3DOF rigid-body block (both are acceptable; know what the block does internally).
  • Structure the top level cleanly now: Plant (continuous), Actuators, Sensors, Flight Software (empty for now) — this is the architecture the whole course builds on.
  • Verify the model with sanity checks rather than vibes.
Primer This is the day Phase 1's math becomes a running model — and the day you set the architecture the rest of the course inherits. Structure the top level into four subsystems now, even though three of them are nearly empty: Plant (the continuous rigid-body dynamics you derived Day 4), Actuators, Sensors, and Flight Software. Building this separation on day one — instead of one tangled diagram — is what lets Phase 2 insert a controller, sensor noise, and an estimator without a rewrite, and what makes the Phase 3 hardware port an honest like-for-like comparison instead of a fresh reimplementation. Equally important today is the discipline of verification: a model that "looks right" is not the same as a model you've proven right. You'll run three specific checks — vertical ascent with zero gimbal deflection should match a hand-propagated 1D trajectory; a fixed small gimbal angle should produce a constant, calculable angular acceleration while thrust is on; zero thrust should produce a clean ballistic arc. Only after those pass do you run the instability demonstration — a 2° initial offset with no controller, tumbling — which becomes Exhibit A at MR-1 for why the rest of this course exists.

Concept 1 — Two ways to build the plant, and why you should know both

You have two legitimate routes to the same 3DOF dynamics, and the objective says either is acceptable — but you should understand what each is doing:

  • From primitives: lay down integrator blocks and MATLAB Function blocks and wire up the six first-order ODEs from Day 4 by hand. More work, but nothing is hidden — every signal is one you named, and when something's wrong you can see exactly where. Recommended for a first build precisely because it forces you to confront your own equations.
  • Aerospace Blockset 3DOF block: a pre-built rigid-body integrator that takes forces and moments in and puts states out. Faster and battle-tested, but it's a black box until you read its docs — you must know its frame conventions, its state ordering, and its sign conventions, or you'll feed it torque with the wrong sign and get a plausible-looking wrong answer.

The instruction "know what the block does internally" is the real assignment here. Whichever route you pick, you should be able to state what integrates to what. A good exercise: build it both ways for the vertical-ascent case and confirm they agree — if they don't, one of them encodes a convention you got wrong, and finding that now is far cheaper than finding it after a controller is wrapped around it.

Concept 2 — Why the four-subsystem architecture, and what goes in each

The single most consequential decision today isn't the dynamics — it's the top-level structure. Splitting into four subsystems now, while three are nearly empty, pays off for the rest of the course:

  • Plant (continuous). The rigid-body dynamics from Day 4. This is the "real world" — it runs in continuous time and it's the one thing that will not exist on the hardware (the real rocket is the plant). Everything else is software or interface.
  • Actuators. Right now, a pass-through: commanded gimbal angle in, actual gimbal angle out. Later it holds servo dynamics (rate limit, lag) and the ±5–10° saturation from Day 4. Isolating it now means adding realistic servo behavior later is a one-subsystem change.
  • Sensors. Right now, ideal: true state in, measured state out. Later (Day 6) it adds noise, bias, and quantization to model the real IMU and barometer. Keeping it separate is what lets you toggle "perfect sensing" on and off to isolate estimator bugs.
  • Flight Software. Empty today. This is the only subsystem that ports to the microcontroller in Phase 3, so everything you eventually want running on hardware — controller, estimator, guidance — goes here and nowhere else. Keeping it walled off is what makes the Phase 3 port an honest like-for-like comparison instead of a reimplementation.

The organizing principle: the boundary between Flight Software and everything else is the hardware boundary. Draw it correctly now and Phase 3 is a port; draw it sloppily and Phase 3 is a rewrite.

Concept 3 — Continuous plant, discrete software (a preview worth setting up now)

The plant is continuous; the flight software will be discrete (fixed sample rate, zero-order hold), because that's how it runs on a microcontroller. You don't implement the discrete side today — Flight Software is empty — but structure the model so that boundary is clean: the interface between Sensors → Flight Software → Actuators is where zero-order holds and a fixed sample rate will live on Day 13. Naming this now means that when you insert the discrete controller later, you're filling a slot you already designed, not surgically splitting a continuous loop. If you're using the Aerospace Blockset, set a fixed-step solver from the start even for the all-continuous model — it makes the later discretization far less disruptive than switching solvers mid-course.

Concept 4 — Verification: proving it, not eyeballing it

The heart of a milestone day is that "looks plausible" is not evidence. Each verification case isolates one behavior against a number you can compute by hand:

  • (a) Vertical ascent, δ = 0. With zero gimbal and a vertical start, rotation should stay exactly zero and the altitude profile should match a 1D hand-propagation of m·z̈ = T(t) − m·g. This validates your translation math and thrust-curve lookup in isolation. Any pitch drift means a sign or frame error.
  • (b) Fixed δ = 2°, thrusting. Angular acceleration should be constant and equal to T·δ·L / J (with δ in radians) for as long as thrust is roughly constant. Plot θ̈ and check the number against your hand calc — this validates the torque model from Day 4 directly.
  • (c) Zero thrust. Pure ballistic motion — a parabola under gravity. This validates that gravity and integration are correct with the actuation removed.

Only after all three pass do you run the instability demonstration: 2° off vertical, δ = 0, no controller. The vehicle should diverge and tumble — and critically, the rate of divergence should match the open-loop instability you can estimate from the linearized dynamics. That plot is MR-1's Exhibit A: it's the physical, simulated proof of why the entire controls half of the course exists. Running it before verification passes is a trap — a tumble from a bug looks identical to a tumble from real instability, and you can't tell which you're looking at unless the model is already trusted.

Debugging discipline When a verification case fails — and one will — resist the urge to tweak gains or nudge signs randomly. Bisect instead: does case (c), zero thrust, work? If yes, gravity and integration are fine and the bug is in the thrust/torque path. Does (a) work but (b) fail? Translation is fine; the bug is in the rotational/torque branch. Each case is designed to light up a different part of the model, so a failure pattern points at a region. This is the same isolate-the-subsystem habit that the four-block architecture is built to support.

Do

  • Run verification cases: (a) δ = 0, vertical start → altitude profile matches a hand-propagated 1D ascent; (b) fixed δ = 2° → constant angular acceleration ≈ T·δ·L/J while thrusting; (c) no thrust → ballistic arc.
  • Demonstrate the instability: start 2° off vertical with δ = 0 and show the divergence. This plot is Exhibit A at MR-1 for why the vehicle needs a controller.
  • Parameter sweep: how does time-to-tumble change with CG location and moment of inertia? (Longer gimbal arm and higher inertia both slow the divergence — confirm the trend matches intuition.)
  • If you built from primitives, rebuild the vertical-ascent case with the Aerospace Blockset 3DOF block (or vice versa) and confirm they agree. Disagreement exposes a convention error.
Common pitfalls
  • Variable-step solver hiding problems. A variable-step solver can mask a stiff or ill-posed model and makes later discretization awkward. Use a fixed-step solver from the start.
  • Aerospace Blockset conventions. The 3DOF block has specific axis, sign, and state-order conventions. Read them — a sign mismatch on the moment input gives a rocket that self-rights with zero gimbal (physically impossible, silently wrong).
  • Degrees vs radians at the block boundary. Simulink trig and the Aerospace blocks expect radians; a stray-degree gimbal input is the classic first-run bug.
  • Tangling the architecture early. One giant flat diagram feels faster today and costs you a rewrite in Phase 2. Build the four subsystems even while empty.
  • Skipping verification to get to the "fun" instability demo. A tumble from a bug and a tumble from instability look the same. Verify first, then demo.
