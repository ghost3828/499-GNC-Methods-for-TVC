Arduino Foundations & Loop Scheduling

Flight software is a loop that must never be late. Learn to write non-blocking code before touching a single sensor.

DAY11

Objectives

  • Set up the toolchain, blink an LED, and use the serial monitor/plotter as your telemetry window.
  • Internalize the non-blocking pattern: never delay() in flight code. Schedule work off millis()/micros().
  • Build the flight-software skeleton: a main loop that runs the GNC task at exactly 100 Hz, mirrors your Simulink Flight Software subsystem (read → estimate → control → command → log), and measures its own timing.
  • Connect the concept: the fixed loop period on the Arduino is the sample time of your discrete controller — the ZOH from Day 7 made real.
Primer Everything since Day 7 has been designed for this moment: your Simulink flight software is already discrete, already sampled at a fixed rate through zero-order holds, which means porting it to a microcontroller is a translation, not a redesign. But embedded code has one hard rule that Simulink enforces for you automatically and C does not: the loop must never be late. Any function that blocks — most notoriously delay() — steals time from the loop and turns your fixed 100 Hz sample rate into a jittery, effectively-lower rate, silently eroding exactly the stability margin you spent Day 7 characterizing. Today is entirely about the discipline of avoiding that trap: scheduling work against millis()/micros() timestamps instead of blocking calls, so multiple things can happen at different rates without ever waiting. The skeleton you build — read → estimate → control → command → log, running at a measured, held 100 Hz — is a direct, function-for-function mirror of the Flight Software subsystem from Simulink; today it just runs stub logic. Every remaining day of Phase 3 fills in one of those functions for real, so getting the scaffolding solid and provably on-rate now is what keeps Days 12 through 14 from turning into timing debugging sessions.

Concept 1 — Why delay() is the enemy

In introductory Arduino code delay(1000) is everywhere, and it's fine there. In flight code it's disqualifying, and understanding why is the whole point of today. delay() is blocking: for its entire duration the processor does nothing else — it can't read a sensor, update the estimator, or command the servo. Your control loop is a contract to do all of those things every 10 ms; a single blocking call breaks the contract.

The consequence is subtle because it's not a crash — it's jitter. If your loop usually takes 6 ms but occasionally a blocking call stretches it to 14 ms, your sample rate isn't a clean 100 Hz anymore; it's a wandering rate that dips whenever something blocks. And recall from Day 7 exactly what an irregular, lower sample rate does: it adds phase lag and eats stability margin. A loop that jitters is a loop that can go unstable on the bench for reasons that have nothing to do with your control design. The insidious part is that it works fine in casual testing and fails only under load — which is why the discipline has to be structural, not something you remember to check.

Concept 2 — The non-blocking pattern: schedule against a clock

The replacement for "wait, then act" is "check the clock, act if it's time, otherwise move on." Instead of blocking, you record when each task last ran and compare against the current time from millis() (milliseconds) or micros() (microseconds):

  • Read the clock, don't sleep against it. Each pass through loop(), ask "has 10 ms elapsed since the last GNC cycle?" If yes, run the cycle and update the timestamp; if no, fall through and let other work happen. Nothing ever waits.
  • Multiple rates coexist. Because tasks are scheduled independently, the GNC cycle can run at 100 Hz, telemetry at 20 Hz, and a status LED at 2 Hz, all from the same loop, none blocking the others. This is exactly the multi-rate structure your sensors will need (the barometer is slower than the IMU, Day 8).
  • Prefer micros() for the control cycle. At 100 Hz your period is 10 000 µs; millisecond resolution is coarse enough to introduce visible timing quantization, so time the GNC loop in microseconds. Watch for millis()/micros() rollover in long runs, though for a short flight it rarely bites.

The mental shift: flight software is cooperative. No single task is allowed to monopolize the processor, because every task depends on the loop coming back around on time. This is the poor-man's real-time scheduler, and for a single 100 Hz control loop it's entirely sufficient.

Concept 3 — The skeleton mirrors Simulink, function for function

The structure you build today isn't arbitrary — it's the Flight Software subsystem from Day 5, translated into C functions. The GNC cycle is five steps in fixed order:

  • read() — pull the latest sensor samples (stubbed today, real on Day 12).
  • estimate() — run the attitude estimator (stub today, the gyro-quaternion propagation from Day 9 on Day 13).
  • control() — the discrete PID (stub today, the ported Day 7 controller on Day 14).
  • command() — write the actuator output (stub today, the servo drive on Day 14).
  • log() — record telemetry for post-run analysis.

Keeping this one-to-one correspondence with the Simulink subsystem is what makes the whole Phase 3 port honest and debuggable: when the bench behaves differently from sim, you can compare function to block directly. Fill each stub in on its scheduled day, and the skeleton never changes shape — you're only ever swapping stub logic for real logic inside a scaffold that already holds rate.

Concept 4 — The loop period IS your sample time

This is the conceptual keystone of the day: the fixed period of your Arduino loop is not merely like the discrete controller's sample time — it is that sample time, the Day 7 zero-order hold made physical. In Simulink the ZOH was a block that held the controller output constant between ticks; on hardware, the loop running once every 10 ms and holding the servo command until the next cycle is literally the same behavior. This means two things: the sample rate you validated in sim (and swept to find its breaking point on Day 7) is only real if the loop actually holds that rate, and any jitter is a discrepancy between the sim's perfect clock and the hardware's real one. That's precisely why the skeleton must measure and report its own timing — worst-case loop duration and overrun count — from day one. You can't just assume you're at 100 Hz; you prove it, because everything downstream was tuned assuming it's true.

Instrument the loop from the start Build timing telemetry into the skeleton before you add any real function, and keep it there for the rest of the course. At minimum, track: actual loop period (should sit tight at 10 000 µs), worst-case execution time per cycle (how much headroom you have before you overrun), and an overrun counter (cycles that took longer than the budget). When you add a sensor read on Day 12 or the estimator on Day 13, these numbers tell you instantly whether the new code fits in the budget. A loop that silently slips from 100 Hz to 70 Hz as you add features is the single most common Phase 3 failure, and this instrumentation catches it the moment it starts — long before it shows up as mysterious controller misbehavior.

Do

  • Work through Blink Without Delay, then extend it: three LEDs at three unrelated rates, no delay().
  • Write fsw_skeleton.ino: a 100 Hz scheduler with stubbed estimate/control functions, printing loop rate and worst-case overrun. Prove it holds rate while printing telemetry.
  • Stress-test the scheduler: deliberately add a slow operation (a long print, a fake computation) and watch the timing telemetry catch the overrun. Then fix it — this rehearses the exact debugging you'll do for real later.
  • Notebook entry: map each Simulink Flight Software block to its function in the skeleton, so the correspondence is documented before the functions get filled in.
Common pitfalls
  • Any delay() in the loop. Even a small one. It blocks everything and jitters your rate. There is no acceptable delay() in flight code.
  • Blocking library calls in disguise. Some sensor libraries block while waiting for data. A blocking read is a hidden delay() — check data-ready flags instead (Day 12).
  • Assuming the rate instead of measuring it. "It should be 100 Hz" is not "it is 100 Hz." Instrument and verify.
  • Heavy work inside the cycle. A big Serial.print or a floating-point-heavy stub can blow the 10 ms budget. Measure execution time and keep headroom.
  • Ignoring micros() rollover. Rare on short flights, but use the subtraction idiom (now - last) that survives rollover rather than comparing absolute times.
